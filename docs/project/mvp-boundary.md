1. MVP 必做

---------

### 1.1 用户与权限

* 用户注册 / 登录

* 用户拥有自己的知识库

* 支持知识库共享访问

* 权限做简单版：拥有者 / 被共享访问者

### 1.2 知识库管理

* 创建知识库

* 查看知识库列表

* 查看知识库详情

* 进入知识库问答

### 1.3 文件处理

* 支持上传 PDF / Markdown / DOC / DOCX

* 文件类型与大小校验

* 基础重复上传控制

* 文件状态跟踪

### 1.4 OCR 与解析

* 文本型文档直接解析

* 扫描件走 OCR

* OCR、解析、入库全部异步执行

* 失败任务可记录原因

### 1.5 问答能力

* 基于指定知识库问答

* 前端流式输出

* 必须带来源片段

* 未命中时优先拒答

### 1.6 工程能力

* Java API 层统一承接前端请求

* 对接 Dify API

* Redis 做缓存 / 限流 / 任务状态加速

* PostgreSQL 存业务数据

* 有基础日志和异常处理
2. 明确不做

-------

本轮明确不做以下内容：

1. 视频链接导入  
   包括 B 站、YouTube 链接解析、字幕抓取、cookies.txt 支持，全部腰斩，不进入本轮范围。

2. 复杂权限系统  
   不做组织、角色树、细粒度资源权限。

3. 企业级多租户

4. 多模型路由平台

5. 向量检索自研平台

6. 自动化评测系统

7. 评论、点赞、协作编辑

8. 复杂搜索中心

9. 独立大而全管理后台

10. 微服务拆分

3. 后续可能做但本轮不做

-------------

* 网页链接抓取入库

* 视频字幕转知识

* 更细粒度共享权限

* 文档标签 / 文件夹

* 引用高亮定位

* 问答质量评测

* 操作审计面板

* 多模型 / 多 Embedding 提供商切换

* 管理后台增强版
4. 容易过度设计的点

-----------

### 4.1 一开始就做全源摄取

首轮只做文档型输入最稳。不要为了“看起来厉害”把视频、网页、API 同步源一起上。

### 4.2 把 QPS 100 误解成所有链路同步承压

OCR、解析、入库都是重任务，应该异步化，不应纳入同步请求 QPS 口径。

### 4.3 同时做 Dify 集成与自研 RAG

首轮用 Dify 做核心即可，别再自研完整知识检索平台。

### 4.4 过早拆微服务

当前最佳策略是**单体 + 清晰模块 + 异步任务**，而不是拆服务堆复杂度。

### 4.5 过度追求权限完备

你当前项目更重要的是主链路闭环和工程稳定性，而不是做一套复杂 ACL 系统。
