1. 项目一句话定义

----------

一个**基于 Dify 作为知识库与 RAG 核心、由 Java 工程层承接多用户接入、文件处理、OCR、异步任务与流式问答的 AI 知识库系统**，面向文档型知识沉淀场景，重点强调**低幻觉、工程稳定性和可讲述的 AI 工程化能力**。  
Dify 适合作为知识库与应用 API 的核心，支持流式输出与知识内容管理。

2. 目标用户

-------

### 2.1 核心目标用户

* 需要将资料沉淀为可问答知识库的**学生、工程师、个人知识管理用户**

* 有轻量共享需求的**小团队内部用户**

### 2.2 用户特征

* 资料以 **PDF、Markdown、DOC、DOCX** 为主

* 需要上传后快速查询，而不是手动阅读全部资料

* 更关注回答是否**基于资料来源**，而不是模型是否“回答得像”
3. 核心问题

-------

当前用户在知识使用上存在以下痛点：

### 3.1 知识接入成本高

不同资料格式不统一，扫描件无法直接检索，资料沉淀效率低。

### 3.2 问答可信度不足

大模型容易脱离资料自由补全，导致回答不可靠。

### 3.3 工程方案常停留在 Demo

很多 AI 项目只有对话壳子，没有处理：

* 多用户接入

* 文件入库链路

* OCR 异步化

* 流式输出

* 并发控制

* OOM 风险治理
4. 为什么现在做

---------

### 已知事实

* Dify 已具备知识库、知识管线、API 集成和流式响应能力，适合作为 AI 应用的底层能力中心。

* Dify 文档提取器对 PDF、DOCX、Markdown 等文本型文档支持较好；图片与扫描类内容通常需要额外 OCR 能力补足。

### 项目价值

* 能形成完整的**AI 工程化项目叙事**

* 能体现你在 **RAG、异步处理、稳定性设计、系统边界控制**上的能力

* 相比普通 CRUD，更适合写进简历并支撑面试深挖
5. 核心场景

-------

### 场景 1：上传文档并入库

用户上传 PDF / Markdown / DOC / DOCX，系统完成校验、解析、OCR、入库处理。

### 场景 2：基于知识库问答

用户针对指定知识库发问，系统基于检索到的资料片段生成回答，并返回引用来源。

### 场景 3：共享知识库访问

知识库创建者可将知识库共享给其他用户，其他用户在权限范围内进行问答和查看。

### 场景 4：任务状态可感知

用户能看到文件当前处于：

* 待处理

* OCR 中

* 解析中

* 入库中

* 已可问答

* 失败
6. 输入 / 输出

----------

6.1 输入
------

* 用户账号信息

* 知识库名称、描述

* 文件上传：PDF / Markdown / DOC / DOCX

* 用户问题文本

6.2 输出
------

* 流式问答结果

* 引用来源片段

* 文件处理状态

* 错误原因提示

* 知识库共享后的可访问视图
7. 成功标准

-------

### 7.1 产品成功标准

* 用户可登录系统并创建知识库

* 用户可上传文档并完成入库

* 扫描件存在可运行的 OCR 链路

* 用户能基于知识库进行问答

* 回答必须展示来源片段

* 当知识不足时系统优先拒答，而不是强行回答

* 支持知识库共享访问

* 前端具备打字流输出体验

### 7.2 工程成功标准

* 文件解析、OCR、入库链路异步化

* 重任务不阻塞主请求线程

* 有基础限流、日志与失败重试机制

* API 层在**轻请求 / 问答入口**维度具备向 **QPS 100** 目标靠拢的设计思路

* 避免因大文件、长文本、OCR 堆积导致 OOM
8. 第一阶段目标

---------

第一阶段目标是做成一个**可演示、可写简历、可继续扩展**的工程化 MVP，覆盖以下闭环：

1. 用户登录

2. 知识库创建与列表管理

3. 文件上传

4. OCR / 解析 / 入库异步任务

5. Dify 知识库接入

6. 基于知识库的低幻觉问答

7. 前端流式输出

8. 简单知识库共享

9. 基础限流、监控与错误处理
